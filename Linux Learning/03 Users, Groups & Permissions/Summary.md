# Multi-user Operating System

Linux از ابتدا به عنوان یک سیستم‌عامل چندکاربره (Multi-user) طراحی شده است.

یعنی چندین کاربر می‌توانند هم‌زمان از یک سیستم استفاده کنند، بدون اینکه اطلاعات یکدیگر را مشاهده یا تغییر دهند.

---

# User چیست؟

هر شخص یا سرویس که وارد سیستم می‌شود، یک User محسوب می‌شود.

نمونه‌ها:

- mehdi
    
- sina
    
- mysql
    
- www-data
    
- postgres
    

تمام کاربران الزاماً انسان نیستند؛ بسیاری از سرویس‌ها نیز با User اختصاصی اجرا می‌شوند.

---

# Root چیست؟

Root قدرتمندترین کاربر Linux است.

UID آن همیشه:

```text
0
```

است.

Root تقریباً هیچ محدودیتی ندارد.

می‌تواند:

- هر فایل را بخواند.
    
- هر فایل را حذف کند.
    
- هر Permissionی را تغییر دهد.
    
- هر سرویسی را مدیریت کند.
    
- هر کاربری را ایجاد یا حذف کند.
    

---

# چرا نباید همیشه Root باشیم؟

کار روزانه با Root خطرناک است.

یک دستور اشتباه مانند:

```bash
rm -rf /
```

می‌تواند کل سیستم را از بین ببرد.

به همین دلیل معمولاً با User عادی کار می‌کنیم و فقط هنگام نیاز از sudo استفاده می‌کنیم.

---

# sudo چیست؟

sudo

(Super User Do)

اجازه می‌دهد یک کاربر عادی، یک دستور را با سطح دسترسی Root اجرا کند.

مثال:

```bash
sudo apt update
```

پس از احراز هویت، فقط همان دستور با دسترسی Root اجرا می‌شود.

---

# Group چیست؟

Group مجموعه‌ای از Userها است.

به جای اینکه Permission را برای تک‌تک کاربران تعریف کنیم، آن را برای Group تعیین می‌کنیم.

نمونه:

- sudo
    
- adm
    
- docker
    
- www-data

# UID چیست؟

User ID

شناسه عددی هر کاربر.

مثال:

```text
0      Root

1000   اولین کاربر عادی

1001   دومین کاربر
```

Kernel کاربران را با UID می‌شناسد، نه Username.

---

# GID چیست؟

Group ID

شناسه عددی Groupها.

---

# Home Directory

هر کاربر فضای اختصاصی خود را دارد.

مثال:

```text

```

کاربر معمولاً فقط داخل Home خود فایل ایجاد می‌کند.

# Permission چیست؟

Linux سه سطح دسترسی دارد:

- Read
    
- Write
    
- Execute


# Best Practices

- همیشه با User عادی کار کنید.
    
- فقط هنگام نیاز از sudo استفاده کنید.
    
- به فایل‌های حساس Permission اضافه ندهید.
    
- Passwordها هرگز داخل فایل‌های معمولی ذخیره نشوند.
    
- قبل از تغییر Permission، وضعیت فعلی را بررسی کنید.
    

---

# Common Mistakes

- استفاده دائمی از Root.
    
- دادن Permission 777 به همه فایل‌ها.
    
- حذف اشتباهی Permissionهای سیستمی.
    
- تغییر Owner فایل‌های سیستم بدون دلیل.
    
- اشتباه گرفتن Permission فایل با Directory.
    

---

# Production Tips

- برای فایل‌های تنظیمات معمولاً Permissionهای محدود مانند **640** یا **644** مناسب هستند.
    
- برای کلیدهای خصوصی SSH معمولاً از Permission **600** استفاده می‌شود.
    
- از **777** فقط در شرایط بسیار خاص و موقت استفاده کنید؛ در محیط‌های Production تقریباً هیچ‌گاه توصیه نمی‌شود.
    
- همیشه قبل از اجرای دستورهای `chmod` یا `chown` روی مسیرهای مهم، یک بار با `ls -l` وضعیت فعلی را بررسی کنید.
    
- اصل **Least Privilege** را رعایت کنید: هر کاربر فقط به حداقل دسترسی موردنیاز خود دسترسی داشته باشد.
